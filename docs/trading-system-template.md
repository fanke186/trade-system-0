# 交易系统模板

> 本模板是 `trade-system-0` 的交易系统 Markdown 骨架定义。
> 用途：把一个人的交易方法整理为可阅读、可追问、可验证、可执行、可版本化的单一事实源。
> 适用范围：趋势交易、价值投资、产业周期、事件驱动、超短、龙头战法、量化规则等。模板风格无关，但每个具体系统必须选择清晰的主逻辑。

## 模板总则

### 什么是交易系统

交易系统是一套决策协议：在明确的市场、周期、资金约束和风险预算下，基于可说明的机会假设，把数据和材料转化为观察、入选、评分、入场、仓位、离场、复盘和迭代规则。

一个完整交易系统至少回答 8 个问题：

1. 我交易什么，不交易什么？
2. 我试图捕捉哪类机会，为什么这类机会可能存在？
3. 我需要哪些数据和证据，哪些证据当前不可用？
4. 什么情况下一个标的进入观察或被排除？
5. 如何把机会质量转成评分或决策档位？
6. 何时入场、买多少、如何处理成交失败和滑点？
7. 何时离场，亏损、盈利、失效和极端情况分别怎么处理？
8. 如何记录、验证、复盘和修改系统，而不是事后随意解释？

### 整体结构

```text
第一层：边界与假设（第 1-4 章）     定义系统要解决什么问题，以及可用证据
第二层：概念与分析（第 5-7 章）     定义语言、市场状态和分析模块
第三层：机会与决策（第 8-10 章）    定义标的池、评分和交易计划
第四层：风险与执行（第 11-12 章）   定义仓位、组合、止损、止盈和熔断
第五层：验证与迭代（第 13 章）      定义记录、复盘、回测、案例和版本治理
```

章节依赖：

- 第 5-12 章使用的术语必须在第 5 章定义。
- 第 8-10 章必须依赖第 3 章机会假设和第 4 章数据能力。
- 第 10-12 章必须形成闭环：每条入场规则都有仓位规则、失效规则和离场规则。
- 第 13 章只允许基于记录和验证修改系统，不允许用单笔结果直接重写规则。

### 填写约定

- 精确优先：能量化就量化，不能量化则写清判断流程、证据来源和人工确认点。
- 边界优先：负面清单、一票否决、能力不足，比泛泛描述更重要。
- 主逻辑优先：一个系统可以有辅助模块，但必须说明主导逻辑和冲突优先级。
- 证据优先：Agent 输出必须追溯到本 Markdown 或本地 K 线证据，不得自行扩展规则。
- 风险优先：没有仓位、止损、最大亏损和熔断规则的系统，不可用于正式评分。
- 合规优先：模板用于整理规则和生成复盘计划，不构成收益承诺或实盘投资建议。

## AI 填写与缺口检测规则

### 填写流程

1. 从用户材料、对话、旧版 Markdown 中提取可填字段。
2. 原文明确支持的内容直接填入。
3. 可合理推断的内容以 `[推断]` 标记，并等待用户确认。
4. 材料没有的信息以 `[待填写]` 标记。
5. 与当前应用能力不匹配的信息以 `[能力不足]` 标记。
6. 存在冲突的信息以 `[冲突]` 标记，并列出冲突来源。

### 缺口严重级别

| 级别 | 标识 | 定义 | Agent 行为 |
|------|------|------|------------|
| 严重缺失 | `[待填写]` | 影响能否交易、评分或风控的关键字段为空 | 阻止正式评分或开仓计划 |
| 不完整 | `[待完善]` | 有方向但不够精确，无法稳定复现 | 可输出观察结论，必须提示缺口 |
| 能力不足 | `[能力不足]` | 系统需要的数据或能力当前 MVP 不支持 | 不得伪造证据，要求用户补材料或降级分析 |
| 冲突 | `[冲突]` | 不同章节规则互相矛盾 | 阻止相关结论，要求用户选择 |
| 已验证 | `[已验证]` | 用户确认且可追溯 | 可用于评分和计划 |

### 完整性检查

AI 在生成或保存交易系统前必须检查：

1. **边界完整**：市场、标的、周期、风格、负面清单是否明确。
2. **目标风险一致**：收益目标是否与风险预算、杠杆、持仓周期矛盾。
3. **机会假设明确**：系统是否说明自己捕捉什么机会，以及何时失效。
4. **术语一致**：第 6-12 章所有术语是否在第 5 章定义。
5. **数据匹配**：评分和信号所需数据是否在第 4 章声明并被当前应用支持。
6. **决策闭环**：每条入场规则是否有对应仓位、止损、止盈或失效规则。
7. **风险闭环**：是否有单笔、单日、组合和回撤层面的限制。
8. **输出可追溯**：评分、结论、计划是否能引用章节规则或本地证据。
9. **验证约束**：回测、模拟或复盘是否有防过拟合和版本治理规则。
10. **风格一致**：主逻辑与辅助逻辑冲突时是否有优先级。

### Agent 输出契约

用于单股评分时，Agent 输出必须包含：

```json
{
  "rating": "focus | watch | reject | data_required | undefined_rule",
  "score": "0-100 或 null",
  "overall_evaluation": "一句话结论",
  "core_reasons": ["每条原因必须引用规则或证据"],
  "risk_points": ["主要风险和失效条件"],
  "trade_plan": {
    "setup": "当前处于什么机会/观察状态",
    "entry": "入场触发条件；不能开仓则写 null",
    "position": "仓位规则；不能开仓则写 null",
    "stop_loss": "初始止损或失效规则；不能开仓则写 null",
    "take_profit": "止盈/减仓/移动止损规则",
    "invalidation": "计划失效条件",
    "no_trade_reason": "不交易原因；可交易则写 null"
  },
  "missing_or_conflicting_rules": ["缺失、冲突或能力不足项"],
  "evidence": ["本地K线、材料或章节引用"]
}
```

禁止项：

- 不得输出收益承诺。
- 不得输出脱离用户确认的实盘买卖指令。
- 不得把未定义规则当成确定结论。
- 不得基于 MVP 未支持数据做确定评分。
- 不得在没有初始止损和仓位规则时输出开仓计划。
- 不得把单笔盈亏作为系统有效或无效的唯一依据。

## 研究依据与设计参考

- 本仓库约束：`README.md` 与 `docs/reference/trend-trader/`，交易系统 Markdown 是单一事实源，评分必须可追溯到系统规则和本地 K 线证据。
- [SEC Investor.gov: Asset Allocation and Diversification](https://www.investor.gov/introduction-investing/getting-started/asset-allocation)：风险承受能力、时间期限、资产配置和分散化是投资计划的基础变量。
- [CFA Institute: Standard III(C) Suitability](https://www.cfainstitute.org/standards/professionals/code-ethics-standards/standards-of-practice-iii-c)：投资行动应与目标、约束、风险承受能力、时间期限和流动性需求一致，并定期复核。
- [CME Group: Risk Management and Your Trade Plan](https://www.cmegroup.com/education/courses/building-a-trade-plan/risk-management-and-your-trade-plan)：交易计划需要明确每笔风险、最大亏损、杠杆和总暴露。
- [FINRA Rule 2270: Day-Trading Risk Disclosure Statement](https://www.finra.org/rules-guidance/rulebooks/finra-rules/2270)：超短和日内交易需要特别重视经验、成本、流动性、保证金和可能超过初始投入的损失。
- [Moskowitz, Ooi, Pedersen: Time Series Momentum](https://pages.stern.nyu.edu/~lpederse/papers/TimeSeriesMomentum.pdf)：趋势/动量类系统应把时间窗口、趋势延续和长期反转风险写清楚。
- [Asness, Moskowitz, Pedersen: Value and Momentum Everywhere](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=1363476)：价值、动量等不同机会来源可以共存，但需要定义冲突处理和组合关系。
- [Bailey et al.: The Probability of Backtest Overfitting](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=2326253)：使用回测或参数优化时，需要样本外验证和防过拟合约束。
