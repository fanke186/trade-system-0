# 《从零构架交易系统》PRD

> 产品代号: `trade-system-0`  
> 版本: 0.1  
> 日期: 2026-05-01  
> 状态: MVP 初稿  
> 参考: `docs/plan/architecture-design.md`、`docs/plan/implementation-guide.md`、`README.md`、[cc-switch](https://github.com/farion1231/cc-switch)、[KLineChart](https://klinecharts.com/)

## 1. 产品定位

《从零构架交易系统》是一个本地桌面应用，帮助用户把自己的交易系统从零整理成一份准确、完整、可执行的 Markdown 文档，并基于这份文档生成专属 AI Agent，用于每日快速复盘、选股评分和交易计划制定。

产品核心围绕“交易系统”而不是“交易执行”。一个交易系统必须能由一个 Markdown 文件完整描述；系统内所有分析、打分、图表标注和交易计划，都必须能追溯到这份 Markdown 文档中的规则。

MVP 聚焦裸 K 交易 + 趋势交易。所需外部信息有且只有历史 K 线（日 K、周 K、月 K），不做分钟级行情、实时盯盘、自动下单、龙虎榜、财报研报等扩展能力。

## 2. 目标用户

| 用户类型 | 主要需求 | MVP 支持方式 |
| --- | --- | --- |
| 裸 K / 趋势交易者 | 每天快速复盘、自选股评分、制定交易计划 | 交易系统 Agent + K 线图 + 股票评分 |
| 正在构建交易系统的个人用户 | 把分散笔记、截图、规则和经验整理为可执行系统 | 上传材料 + 对话澄清 + Markdown 生成 |
| 希望长期迭代系统的用户 | 保留系统版本、复盘记录和规则演化过程 | 交易系统版本化、评分记录可追溯 |
| 后续扩展用户 | 根据交易风格接入超短、基本面或其他数据模块 | 能力包机制预留，MVP 不实现 |

## 3. 核心问题

- 用户的交易系统通常分散在笔记、聊天记录、截图和脑内经验中，缺少一份可执行的单一事实源。
- AI 可以帮助复盘，但如果没有绑定用户自己的交易系统，输出容易泛化、漂移或擅自加规则。
- 每日复盘的核心工作是快速判断股票是否符合自己的系统，而不是泛泛解释股票涨跌。
- 裸 K / 趋势交易最需要的是稳定的历史 K 线、关键位置识别、图表标注和一致的交易计划输出。
- 后续不同交易风格所需数据差异很大，需要从 MVP 开始保留可扩展的原子能力模型。

## 4. MVP 产品目标

### 4.1 核心目标

- 让用户通过上传材料和对话交互，生成一份可版本化的 `my_trade_system.md`。
- 将 `my_trade_system.md` 编译为一个专属 AI Agent，使其只能按该交易系统分析股票。
- 提供原子能力 `(stock_code, my_trade_system) => (打分, 总体评价, 核心原因, 交易计划)`。
- 支持下载单只股票全部历史日 K，并聚合或读取周 K、月 K。
- 提供可交互 K 线图，支持缩放、悬停详情、成交量、成交额、普通/对数坐标，以及横线/射线绘制和高低点吸附。
- 提供类似同花顺的自选股票池，用于批量评分和每日复盘。

### 4.2 成功标准

- 用户能在桌面应用中完成“上传材料 → 对话澄清 → 生成交易系统 Markdown → 创建 Agent → 评分股票 → 查看图表和交易计划”的闭环。
- 同一只股票、同一版本交易系统、同一份 K 线数据下，评分结果结构稳定，核心原因可追溯。
- Agent 不应输出交易系统 Markdown 中没有依据的规则；无法判断时必须标记为“信息不足”或“系统未定义”。
- MVP 在没有分钟行情、实时行情和交易接口的情况下，仍能完成每日复盘与选股。
- 产品以可直接打开的桌面应用交付，不以网页站点为主形态。

## 5. 核心概念

### 5.1 交易系统 Markdown

交易系统 Markdown 是产品的单一事实源。它既是用户可读文档，也是 Agent 分析股票时必须遵守的规则来源。

MVP 推荐结构：

```markdown
# 交易系统名称

## 1. 系统定位
适用市场、交易周期、目标机会类型、明确不做什么。

## 2. 数据需求
需要的行情周期、图表视角、是否需要成交量/成交额。

## 3. 入选条件
趋势结构、K 线形态、关键位置、量价关系、排除条件。

## 4. 评分规则
评分维度、分值权重、加分项、扣分项、最低入选分。

## 5. 交易计划规则
观察条件、入场条件、止损规则、止盈/减仓规则、失效条件。

## 6. 图表标注规则
需要标注的趋势线、关键位、支撑阻力、近期高低点。

## 7. 复盘输出格式
总体评价、核心原因、风险点、交易计划、是否加入观察。

## 8. 案例与反例
符合系统的例子、不符合系统的例子、边界情况。
```

准确和完整的判定标准：

- 入选条件、排除条件、评分规则、交易计划规则不能只停留在口号。
- 每个评分维度必须有可解释的判断依据和默认分值口径。
- 每个交易计划必须能说明入场、止损、止盈、失效或“不交易”的条件。
- 文档未定义的规则，Agent 不得自行补充为确定结论。
- 每次修改生成新版本号和内容哈希，评分记录绑定交易系统版本。

### 5.2 交易系统 Agent

交易系统 Agent 是从 Markdown 编译出来的专家助手。它的职责是按用户定义的交易系统分析股票，而不是提供通用投资建议。

MVP 能力：

- 从 Markdown 提取系统定位、数据需求、评分规则、交易计划规则和输出格式。
- 基于上传材料和对话问答补齐缺口，生成或更新 Markdown。
- 在评分股票时调用 K 线数据和图表标注能力。
- 输出结构化评分、核心依据、交易计划和不确定性说明。
- 支持模型配置，MVP 优先支持 DeepSeek 和 ChatGPT / OpenAI API。

约束：

- Agent 必须引用交易系统版本。
- Agent 必须区分“系统规则判断”和“模型推断”。
- 当 K 线数据不足或交易系统规则缺失时，不能强行打高分。
- Agent 不负责实盘下单、收益承诺或自动交易。

### 5.3 原子能力

MVP 所有上层功能都围绕三个原子能力组合：

| 原子能力 | 输入 | 输出 |
| --- | --- | --- |
| 交易系统生成 | 上传文件、对话记录、现有 Markdown | `my_trade_system.md`、缺口问题、版本记录 |
| 股票评分 | `stock_code`、`trade_system_id/version`、K 线数据 | 分数、总体评价、核心原因、交易计划、图表标注 |
| K 线支撑 | 股票代码、周期、日期范围 | 日/周/月 K、成交量、成交额、图表数据 |

## 6. MVP 功能范围

### 6.1 交易系统构建

| 功能 | MVP 要求 |
| --- | --- |
| 材料上传 | 支持上传 `.md`、`.txt`，优先支持可提取文本的 `.pdf`；图片识别后续再做 |
| 对话澄清 | Agent 根据材料缺口追问，例如交易周期、入选条件、止损规则、排除条件 |
| Markdown 生成 | 生成一份完整 `my_trade_system.md`，用户可编辑、保存、导出 |
| 完整性检查 | 标记缺失章节、互相矛盾的规则、无法落地的评分项 |
| 版本管理 | 每次保存生成版本号、更新时间、内容哈希和变更摘要 |
| Agent 创建 | 从指定版本 Markdown 创建或更新专属交易系统 Agent |

### 6.2 股票评分

原子能力：

```text
(stock_code, my_trade_system) => (score, overall_evaluation, core_reasons, trade_plan)
```

MVP 默认输出结构：

```json
{
  "stock_code": "002261",
  "trade_system_id": "ts_001",
  "trade_system_version": "v3",
  "score": 76,
  "rating": "watch",
  "overall_evaluation": "趋势结构基本符合系统，但距离关键压力位较近。",
  "core_reasons": [
    "周线仍处于上升结构",
    "日线回踩后未跌破关键支撑",
    "近期量能未明显放大，确认度不足"
  ],
  "evidence": [
    {
      "rule": "入选条件/趋势结构",
      "result": "pass",
      "data_ref": "daily:2026-04-30"
    }
  ],
  "trade_plan": {
    "setup": "观察回踩关键支撑后的承接",
    "entry": "放量突破前高后考虑",
    "stop_loss": "跌破最近 swing low",
    "take_profit": "接近上方压力位分批处理",
    "invalidation": "跌破趋势结构或评分低于系统阈值",
    "no_trade_reason": null
  },
  "chart_annotations": [
    {
      "type": "horizontal_line",
      "price": 18.5,
      "reason": "近期关键压力位"
    }
  ],
  "uncertainty": []
}
```

评分规则：

- 默认采用 0-100 分；具体权重优先来自交易系统 Markdown。
- 若 Markdown 未定义分数档位，MVP 默认：80+ 为重点观察，60-79 为观察，60 以下为不符合。
- 必须输出“核心原因”，且原因应对应交易系统中的规则章节。
- 必须允许输出“不交易”，并说明不交易的系统内原因。
- 同一输入下应尽量稳定；模型随机性默认低温度。

### 6.3 K 线信息模块

MVP 时效要求为日级别，不支持分钟线、小时线和实时逐笔。

核心边界：K 线绘制、股票评分、每日复盘都不负责临时下载行情。所有 K 线必须先由数据同步模块写入本地结构化存储，再由查询服务读取给图表和 Agent 使用。

三层能力：

| 层级 | 职责 | MVP 要求 |
| --- | --- | --- |
| 数据同步 | 从外部数据源下载或更新 K 线，并写入本地库 | 支持单只股票全量同步和增量同步，支持交易日判断和数据源降级 |
| 本地结构化存储 | 保存证券信息、交易日历、日/周/月 K | 参考 `docs/plan` 的 `securities`、`trade_calendar`、`bars_1d`、`bars_1w`、`bars_1M` |
| 查询服务 | 从本地库读取 K 线给评分和图表 | 按 `stock_code + frequency + date_range` 查询，不触发下载 |

数据同步要求：

- `sync_kline(stock_code, mode=full|incremental)` 是 K 线下载和更新的唯一入口。
- `full` 模式下载指定股票全部历史日 K，并写入本地结构化库。
- `incremental` 模式基于本地最后交易日，只补齐缺失交易日。
- 同步时先判断交易日历；非交易日更新应返回明确跳过原因。
- 数据源优先级保持可配置；公开数据源不可用时允许降级到 sample 数据用于开发。
- 日 K 写入后，系统聚合生成周 K、月 K；周/月 K 的 `trade_date` 采用该周/月最后一个交易日。

本地结构化存储要求：

- 应用状态、交易系统、Agent、评分记录、自选池使用 SQLite。
- K 线结构化数据推荐使用 DuckDB；数据量增长后可扩展为 DuckDB + Parquet 分区。
- 证券基础信息独立存储，至少包含股票代码、名称、交易所、上市状态。
- 交易日历独立存储，至少包含交易日期和是否开市。
- 日 K 至少包含：日期、开盘、最高、最低、收盘、前收盘、成交量、成交额、复权因子。
- 周 K、月 K 至少包含：周期最后交易日、开盘、最高、最低、收盘、成交量、成交额。
- 本地写入必须按股票和日期去重，避免重复同步污染图表和评分。

查询服务要求：

- `get_bars(stock_code, frequency=1d|1w|1M, start_date, end_date, limit)` 只读取本地结构化库。
- `get_data_coverage(stock_code)` 返回该股票日/周/月 K 的起止日期、最后更新时间和缺失状态。
- 查询返回为空或覆盖不足时，调用方显示“数据未就绪/请先同步”，不得在图表或评分组件内隐式下载。
- 图表输入固定为本地查询返回的结构化 bars：`date/open/high/low/close/volume/amount`。
- 股票评分可以额外读取 `pre_close`、`turnover`、`adj_factor` 等字段，但核心评分不得依赖未入库数据。

### 6.4 自选股票池

MVP 提供类似同花顺自选股的股票池视图，但只要求日级别复盘，不要求实时行情。

能力要求：

- 创建、重命名、删除股票池。
- 添加、移除股票，支持股票代码和名称展示。
- 展示最近交易日收盘价、涨跌幅、成交额、最近评分、评分时间、系统版本。
- 支持对整个股票池批量评分，并按分数排序。
- 点击股票进入 K 线详情和评分详情。
- 支持标记“重点观察”“暂不关注”“已排除”等本地状态。

### 6.5 图表绘制能力

图表能力借鉴开源项目 KLineChart。MVP 使用 KLineChart 作为前端图表基础，运行在桌面应用的 WebView 中。

图表组件只消费 `get_bars(...)` 返回的本地结构化数据。打开图表时如果本地 K 线缺失或覆盖不足，只显示“数据未就绪/请先同步”，并提供跳转到数据同步页面的操作；图表组件本身不发起下载。

必须支持：

- K 线绘制：日 K、周 K、月 K 切换。
- 缩放和平移：鼠标滚轮、拖拽、时间范围缩放。
- 悬停详情：显示单根 K 线的日期、开盘、最高、最低、收盘、涨跌幅、成交量、成交额。
- 成交量和成交额：主图下方至少支持成交量副图，成交额可作为副图或 tooltip 字段。
- 坐标类型：普通坐标和对数坐标切换。
- 画线工具：横线、射线。
- 自动吸附：画线端点吸附到附近 K 线最高点或最低点。
- Agent 标注：评分结果中的关键位、趋势线、风险位可以自动渲染到图表。
- 手动标注：用户手动画线后可保存到该股票和交易系统版本的复盘记录中。
- 断网可用：只要本地库已有数据，图表在断网状态下仍应正常渲染、缩放、悬停和画线。

暂不支持：

- 分钟线、盘口、分时图。
- 复杂画线工具，如矩形、斐波那契、通道线。
- 多股票同屏联动。

## 7. 信息架构

应用采用 cc-switch 类桌面布局：左侧模块导航，中间主工作区，右侧上下文或详情面板，顶部展示当前交易系统、模型 Provider 和数据状态。

| 模块 | 页面 | MVP 作用 |
| --- | --- | --- |
| 首页 / 每日复盘 | Daily Review | 选择交易系统和股票池，一键更新 K 线并批量评分 |
| 交易系统 | Trade System | 上传材料、对话澄清、编辑 Markdown、版本管理 |
| Agent | Agent | 查看由交易系统生成的专家 Agent，选择模型并测试问答 |
| 股票评分 | Stock Review | 单只股票评分、核心原因、交易计划、图表标注 |
| K 线图表 | Chart | K 线查看、缩放、画线、坐标切换、标注保存 |
| 自选股票池 | Watchlist | 股票池维护、批量评分、排序筛选 |
| 数据 | Data | 下载/更新单只股票历史 K 线，查看数据覆盖范围 |
| 设置 | Settings | DeepSeek、ChatGPT / OpenAI、数据源和本地存储设置 |

## 8. 核心用户流程

### 8.1 从材料生成交易系统

1. 用户打开桌面应用，进入“交易系统”页面。
2. 上传已有交易笔记、规则文档或复盘材料。
3. Agent 读取材料并生成初版 Markdown。
4. 系统标记缺失规则，例如止损、评分权重、排除条件。
5. 用户通过对话补充缺口。
6. 系统生成 `my_trade_system.md`，用户确认并保存版本。
7. 应用基于该版本创建专属交易系统 Agent。

验收标准：

- 能从材料生成结构化 Markdown。
- 能列出未定义或互相矛盾的规则。
- 保存后能看到版本号、哈希和变更摘要。

### 8.2 单只股票评分

1. 用户输入股票代码或从股票池点击股票。
2. 系统调用 `get_data_coverage(stock_code)` 检查本地 K 线覆盖范围。
3. 本地数据缺失或覆盖不足时，系统引导用户执行 `sync_kline(...)`，评分流程暂停，不隐式下载。
4. 用户选择交易系统版本。
5. Agent 通过 `get_bars(...)` 读取本地日/周/月 K，并调用图表分析能力。
6. 系统返回分数、总体评价、核心原因和交易计划。
7. 图表基于同一份本地 bars 自动展示关键位、趋势射线、止损位等标注。

验收标准：

- 评分输出必须包含四个核心字段：打分、总体评价、核心原因、交易计划。
- 每条核心原因能对应交易系统规则或 K 线证据。
- 数据不足时不能给出伪确定结论，也不能在评分流程中偷偷下载。

### 8.3 每日快速复盘和选股

1. 用户进入首页 / 每日复盘。
2. 选择交易系统版本和自选股票池。
3. 点击“更新 K 线并评分”。
4. 系统先对股票池内股票执行 `sync_kline(..., mode=incremental)`，将缺失日 K 写入本地库并聚合周/月 K。
5. 同步完成后，评分任务只读取本地 K 线库，批量调用评分能力。
6. 列表按分数排序，展示评级、核心原因摘要和计划摘要。
7. 用户点击高分股票查看图表与完整交易计划，图表读取同一份本地结构化数据。

验收标准：

- 股票池批量评分可完成并展示成功/失败状态。
- 每只股票评分记录可回溯到交易系统版本、本地 K 线日期和同步时间。
- 用户能快速筛出重点观察标的。

### 8.4 图表画线和复盘保存

1. 用户打开股票图表。
2. 系统调用 `get_bars(...)` 从本地库读取日 K、周 K 或月 K。
3. 若本地数据未就绪，图表显示同步提示，不触发下载。
4. 用户切换日 K、周 K、月 K。
5. 开启横线或射线工具。
6. 鼠标靠近 K 线最高/最低点时自动吸附。
7. 用户保存标注。
8. 标注进入该股票的复盘记录，并可被后续评分引用。

验收标准：

- 画线位置不会因缩放和平移明显错位。
- 悬停详情准确展示 OHLC、成交量和成交额。
- 普通/对数坐标切换后图表和标注仍可读。
- 断网但本地已有数据时，图表仍可正常读取和渲染。

## 9. 技术架构要求

### 9.1 应用形态

产品不是网页，而是可直接打开的桌面应用。MVP 推荐采用：

- 桌面壳：Tauri 2。
- 前端：React + TypeScript。
- 后端命令层：Rust Tauri Commands。
- 本地数据库：SQLite 作为应用状态 SSOT。
- K 线分析库：DuckDB 或 SQLite 独立表；若数据量增长，优先 DuckDB。
- 图表：KLineChart。
- 模型调用：DeepSeek、ChatGPT / OpenAI，另预留 OpenAI-compatible 自定义 Provider。

采用 cc-switch 风格的原因：

- 桌面应用可直接安装打开，避免本地用户启动前后端服务。
- Provider 配置、启用、测试可以做成清晰的设置页。
- SQLite 单一事实源适合本地配置、交易系统版本和评分记录。
- Tauri IPC 能把前端交互和本地数据、文件、模型调用隔离清楚。

### 9.2 本地数据模型

MVP 需要的核心实体：

| 实体 | 说明 |
| --- | --- |
| `trade_systems` | 交易系统基本信息 |
| `trade_system_versions` | Markdown 内容、哈希、变更摘要、完整性检查结果 |
| `model_providers` | DeepSeek、OpenAI / ChatGPT、自定义 OpenAI-compatible 配置 |
| `agents` | 由交易系统版本生成的 Agent 配置 |
| `stocks` | 股票代码、名称、市场 |
| `securities` | 证券基础信息，包含代码、名称、交易所、上市状态 |
| `trade_calendar` | 交易日历，支撑增量更新和非交易日跳过 |
| `bars_1d` | 本地日 K 结构化数据 |
| `bars_1w` | 由日 K 聚合或同步得到的周 K 数据 |
| `bars_1M` | 由日 K 聚合或同步得到的月 K 数据 |
| `watchlists` | 自选股票池 |
| `watchlist_items` | 股票池明细和本地状态 |
| `stock_reviews` | 单只股票评分结果、交易计划、证据和标注 |
| `chart_annotations` | 用户或 Agent 生成的图表标注 |

K 线实体可放在 DuckDB 中，应用状态实体放在 SQLite 中。图表和评分层不得直接访问外部行情源，只能通过本地查询服务读取这些实体。

### 9.3 能力扩展机制

MVP 只实现 K 线能力包，但架构上需要允许未来按交易系统声明数据依赖。

能力包示例：

| 能力包 | 适用交易系统 | MVP 状态 |
| --- | --- | --- |
| K 线能力 | 裸 K、趋势交易、波段交易 | 必须实现 |
| 超短能力 | 连板梯队、情绪周期、龙虎榜、竞价 | 后续扩展 |
| 基本面能力 | 财报、研报、估值、行业比较 | 后续扩展 |
| 交易执行能力 | 条件单、模拟交易、实盘网关 | 非 MVP |

交易系统 Markdown 的“数据需求”章节应能声明需要哪些能力包。MVP 如检测到非 K 线能力需求，应提示“当前版本不支持该数据能力”，而不是用模型猜测。

## 10. 非功能要求

| 类型 | 要求 |
| --- | --- |
| 本地优先 | 交易系统文档、评分记录、自选池和 K 线结构化库默认存本地 |
| 可追溯 | 每次评分绑定交易系统版本、模型 Provider、K 线数据日期和输出时间 |
| 稳定性 | 评分模型默认低温度，输出结构固定，失败时给出明确错误 |
| 安全性 | API Key 不明文写入普通文档，优先使用系统安全存储或加密本地配置 |
| 可编辑 | Markdown 永远可由用户直接查看、修改、导出 |
| 可降级 | 数据源不可用时，保留本地已有数据；开发环境可使用 sample 数据 |
| 数据边界 | 下载/更新只发生在数据同步模块；评分和图表只读取本地库 |
| 响应速度 | 单只股票评分应适合交互式使用；批量评分需要展示进度和失败项 |
| 图表体验 | K 线缩放、平移、悬停和标注必须流畅，标注与价格轴保持一致 |
| 明确边界 | 不做投资建议承诺，不做收益承诺，不默认执行交易 |

## 11. MVP 验收标准

### 11.1 交易系统与 Agent

- 可上传材料并生成 `my_trade_system.md`。
- 可编辑、保存、导出交易系统 Markdown。
- 可看到完整性检查结果和待补充问题。
- 可基于指定版本创建 Agent。
- Agent 回答时能遵守交易系统边界，缺失规则时明确提示。

### 11.2 模型配置

- 可配置 DeepSeek API。
- 可配置 ChatGPT / OpenAI API。
- 可选择当前启用 Provider。
- 可测试 Provider 连通性。
- 评分和 Agent 对话记录中能看到使用的 Provider。

### 11.3 K 线数据

- 可输入股票代码下载全部历史日 K。
- 可查看日 K、周 K、月 K。
- 可增量更新已有股票数据。
- 可展示数据覆盖范围和最后更新时间。
- 下载/更新任务成功后，本地结构化库可查询日 K、周 K、月 K。
- 周 K、月 K 的聚合口径稳定，使用该周/月最后一个交易日作为周期日期。
- `get_bars(...)` 查询本地数据时不触发外部下载。
- 数据不足时评分流程能明确提示。

### 11.4 图表

- KLineChart 能渲染日/周/月 K。
- 支持缩放、平移、悬停详情。
- 支持成交量和成交额展示。
- 支持普通坐标和对数坐标切换。
- 支持横线、射线绘制，并吸附到 K 线最高/最低点。
- Agent 评分生成的关键位能自动显示在图表上。
- 图表只读取本地 K 线库；本地数据缺失时提示“数据未就绪/请先同步”。
- 断网情况下，只要本地库已有数据，图表仍能正常渲染。

### 11.5 股票评分和每日复盘

- 单只股票评分输出分数、总体评价、核心原因、交易计划。
- 评分结果可保存，并绑定交易系统版本。
- 评分前必须检查本地 K 线覆盖范围；数据不足时提示同步，不使用模型猜测补齐。
- 自选股票池可批量评分。
- 每日复盘批量评分必须先完成 K 线库同步，再读取本地库评分。
- 批量评分列表可按分数排序。
- 用户能从列表进入股票详情、查看图表和完整计划。

## 12. 阶段规划

| 阶段 | 目标 | 交付 |
| --- | --- | --- |
| Phase 0 | 桌面应用基础 | Tauri 壳、React 布局、SQLite、Provider 设置骨架 |
| Phase 1 | 交易系统文档 | 材料上传、Markdown 编辑、版本管理、完整性检查 |
| Phase 2 | Agent 生成 | DeepSeek / OpenAI 配置、交易系统 Agent、对话澄清 |
| Phase 3 | K 线数据 | 单股历史 K 线下载、日/周/月查询、数据覆盖状态 |
| Phase 4 | 图表能力 | KLineChart 集成、缩放、悬停、坐标切换、横线/射线/吸附 |
| Phase 5 | 股票评分 | 单股评分、结构化输出、图表标注、评分记录 |
| Phase 6 | 每日复盘 | 自选股票池、批量评分、排序筛选、复盘摘要 |

## 13. 非目标

- MVP 不做网页应用形态。
- MVP 不做分钟线、小时线、盘口、分时和实时行情。
- MVP 不做自动交易、实盘下单、MiniQMT 网关和条件单。
- MVP 不做超短数据，如连板梯队、情绪周期、龙虎榜。
- MVP 不做基本面数据，如财报、研报、估值模型。
- MVP 不承诺自动发现牛股，也不提供投资收益保证。
- MVP 不让 Agent 脱离交易系统 Markdown 自由发挥成通用荐股助手。

## 14. 待确认事项

- 上传材料的第一版是否必须支持 PDF，还是先只支持 Markdown / TXT。
- K 线数据源优先级：是否沿用 trend-trader 的 QUANTAXIS → 东方财富/腾讯 → sample 降级路径。
- 交易系统 Markdown 是否需要固定模板强校验，还是允许用户自定义章节后由 Agent 映射。
- 评分是否必须全自动，还是允许用户手动确认关键线位后再生成最终分数。
- API Key 存储是否使用系统 Keychain / Credential Manager，还是先使用加密本地配置。
- 自选股票池是否需要导入同花顺/通达信导出的股票列表。
